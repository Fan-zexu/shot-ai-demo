# MVP 验证说明

验证分为两层：自动化工程证据回答“系统是否按契约工作”，真实样本实验回答“受控投篮视频上的精度和表达是否达到目标”。前者不能替代后者。

## 自动化工程证据

在干净工作区执行：

```bash
pnpm install --frozen-lockfile
python3.11 -m venv .venv
.venv/bin/pip install -r services/pose-worker/requirements.lock
pnpm verify
```

| 范围 | 自动检查 | 通过标准 |
| --- | --- | --- |
| 公共契约 | TypeBox 闭合结构、六事件顺序、Python 夹具互认 | 全部通过，无未知字段 |
| 视频与姿态 | 探测、时间轴、视角、质量门控、事件、归一化、预览、安全路径 | Python 测试零失败 |
| 比较引擎 | 兼容性、事件硬锚点、阶段内受限 DTW、低置信度、差异窗口 | TypeScript 测试零失败 |
| API | 文件哈希/Range、事务状态、恢复、重试、软删除、报告与导出 | 集成测试零失败 |
| H5 | 共享播放状态、六事件、低置信度不高亮、40ms 校正规则 | Vitest 零失败 |
| 浏览器 | 三模式共享状态、320px 双列/44px 控件、桌面调试证据 | Chromium 与 WebKit 零失败 |
| 构建 | 严格 TypeScript 与 Vite production build | 零错误 |

E2E 通过网络拦截注入确定性 `ReportBundle` 和调试摘要，只证明前端对公共数据的消费一致，不证明 MediaPipe 对真实投篮的精度。

报告保存 5 个动作阶段、114 个同步采样和 9 个差异窗口。共同可比较区域为下肢、躯干、投篮手臂和全身时序；引导手因覆盖不足统一标记为不可比较。由于模板和用户视频速度都未经确认，结果阈值快照中的速度权重为 `0`，不输出真实速度、节奏或阶段耗时差异。

本次记录证明这一个真实样本能够走通，不代表 5–8 个有效用户视频的计划已经完成，也不代表六事件相对人工标注的精度已经达标。

### 人工理解记录

每位测试者分别使用并排视频、骨架叠加、动作通道后记录：能否指出主要差异、首次理解耗时、是否误把差异当成错误、是否理解“释放姿态代理”和“参考动作通道”的边界、是否依赖桌面调试数据。MVP 不据此输出评分或训练建议。

## 放行条件

工程 Demo 只有在以下证据同时成立时才可称为通过真实链路验收：

1. `pnpm verify` 在干净环境零失败。
2. 上述真实样本数量达到计划下限，文件哈希和人工标注可追溯。
3. 所有硬指标达标；任何真实离手、评分、诊断或训练收益表述仍保持禁用。
4. 任一报告能从数据库记录追溯到源文件哈希、两份动作产物、模型/算法/阈值版本和任务耗时。

当前已完成自动化工程验证、两个真实模板冒烟和一个真实用户视频纵切；用户视频数量、边界样本、人工事件标注和完整指标尚未达到真实链路放行条件。
